# 프로젝트 상세 설계 기준

## 목적

이 문서는 포트폴리오의 프로젝트 상세 페이지를 어떤 기준으로 설계하고, 어떤 서사로 써야 하는지 정리한 기준 문서다.

방향은 단순하다.

- 인터랙션보다 실무 판단력이 먼저 보이게 한다.
- 예쁜 프로젝트 소개가 아니라, 문제 해결 사례 문서처럼 읽히게 만든다.
- 현재 포트폴리오에 있는 대표 프로젝트는 실제 경력에서 드러난 역량 축과 연결되도록 정리한다.

## 이 포트폴리오에서 보여줘야 하는 핵심 역량

경력 데이터를 기준으로 보면, 상세 페이지는 아래 역량을 반복적으로 증명하는 쪽이 맞다.

### 1. 복잡한 서비스 흐름을 UI로 정리하는 능력

근거:

- 한국방송통신대학교 모바일 시스템의 조건부 동적 폼
- 케이뱅크 4단계 입력 플로우
- Tarome의 선택-결과-채팅 흐름
- 생일24의 모바일 사용자 플로우

### 2. 백엔드 제약 속에서 프론트에서 풀어내는 능력

근거:

- 방송대 프로젝트의 데이터 재가공 구조
- 사내 백오피스 데이터 정합성 처리
- Tarome의 상태 분리와 단계 전이 재설계

### 3. 정보 구조와 우선순위를 재정의하는 능력

근거:

- Chap Landing의 메시지와 CTA 재배치
- Nova Admin의 운영 정보 재구성
- Wello 정책 추천 화면의 조건별 탐색 구조

### 4. 실무형 프론트엔드 구현력

근거:

- Nuxt/Next 기반 화면 구현
- SSR/SEO 대응
- Canvas, Drag & Drop, Chart, Dynamic Form
- 모바일 우선 화면 구성과 운영용 UI 구현

즉, 프로젝트 상세는 "멋진 화면을 만들었다"보다 아래 질문에 답해야 한다.

- 왜 이 구조가 필요했는가
- 어떤 제약이 있었는가
- 어떤 판단을 했는가
- 구현은 어떤 방식으로 풀었는가
- 결과적으로 사용성과 운영성이 어떻게 좋아졌는가

## 추천 상세 구조

프로젝트 상세는 아래 순서로 통일한다.

### 1. Hero

- 프로젝트를 한 줄로 설명하는 강한 문장
- 기간, 역할, 기술
- 썸네일 또는 대표 이미지

이 영역은 시각적으로 강해도 된다.
하지만 여기서 중요한 건 분위기보다 "이 프로젝트가 무엇인지 3초 안에 이해되는 것"이다.

### 2. 프로젝트 한 줄 요약

다음 형식 중 하나로 정리한다.

- `복잡한 흐름을 끊기지 않는 단계형 UX로 재구성한 프로젝트`
- `정보 우선순위를 다시 설계해 운영 판단 속도를 높인 프로젝트`
- `핵심 메시지와 전환 흐름을 한 화면 안에 정리한 랜딩 프로젝트`

### 3. 문제 정의

이 섹션에는 기능 설명이 아니라, 왜 문제가 문제였는지를 적는다.

좋은 형식:

- 사용자가 어디에서 끊겼는지
- 운영자가 무엇을 빠르게 판단하지 못했는지
- 기존 구조가 왜 복잡했는지
- 어떤 제약 때문에 일반적인 방법을 쓰기 어려웠는지

### 4. 제약 조건

현재 데이터 모델에는 없지만, 실제로는 이 섹션이 중요하다.

예시:

- 백엔드 수정 어려움
- 기획서만 있고 실제 API 미완성
- 금융권 QA 대응
- 모바일 화면 우선
- 운영 화면의 정보 밀도 높음

이 섹션이 들어가면 문제 해결의 설득력이 커진다.

### 5. 핵심 판단

이 프로젝트에서 내가 한 중요한 결정을 2~4개 정도만 적는다.

예시:

- 단계를 페이지 분리가 아니라 상태 흐름으로 재설계
- 핵심 CTA를 첫 화면에 집중
- 운영 지표와 상세 정보를 분리
- 데이터 가공 책임을 프론트에서 명시적으로 분리

이 섹션이 가장 중요하다.
실무자는 구현보다 판단을 더 높게 본다.

### 6. 구현 방식

현재의 `implementation`은 이 섹션으로 유지 가능하다.
다만 아래 두 축으로 분리하는 게 더 좋다.

- UI/UX 구현
- 상태/데이터/구조 구현

### 7. 트러블슈팅

여기는 "에러를 고쳤다" 수준이 아니라,

- 왜 초기에 구조가 깨졌는지
- 무엇을 기준으로 다시 분리했는지
- 리팩토링 후 어떤 점이 안정적이 되었는지

를 적는다.

### 8. 결과 또는 배운 점

현재 데이터 모델에는 없지만 추가하는 것이 좋다.

예시:

- 사용자 흐름이 자연스럽게 이어지도록 개선
- 운영자의 시선 이동 비용 감소
- 유지보수하기 쉬운 상태 구조 확보
- 후속 확장이 가능한 구조 확보

정량 지표가 없더라도 괜찮다.
대신 결과를 사용성과 구조 개선 관점에서 서술한다.

## 프로젝트별 상세 방향

## Tarome

### 보여줘야 하는 핵심

- 단계형 UX 설계 능력
- 상태 전이 설계 능력
- 모바일 우선 화면 구조 정리 능력

### 좋은 상세 서사

1. 문제
   결과와 채팅이 분리되면 몰입이 끊긴다.
2. 제약
   단계별 상태가 강하게 얽혀 예외 케이스가 많다.
3. 판단
   화면 단위가 아니라 상태 흐름 단위로 분리해야 한다.
4. 구현
   선택 상태, 결과 상태, 후속 액션 상태를 분리
5. 결과
   선택에서 해석, 후속 대화까지 자연스럽게 이어지는 흐름 형성

### 상세에서 강조할 문장

- "끊어진 화면 전환이 아니라, 하나의 읽히는 흐름으로 재구성했다."

---

## Chap Landing

### 보여줘야 하는 핵심

- 메시지 우선순위 설계
- CTA 배치 감각
- 브랜드 톤과 구조 정리 능력

### 좋은 상세 서사

1. 문제
   첫 화면에서 핵심 메시지가 약하고 전환이 분산됨
2. 제약
   랜딩은 짧은 스크롤 안에 메시지와 전환을 동시에 담아야 함
3. 판단
   가치 제안과 CTA를 같은 흐름 안에 배치
4. 구현
   메시지 계층, 타이포, 간격 시스템 정리
5. 결과
   정보 탐색 비용을 줄이고 행동 유도 흐름을 선명하게 만듦

### 상세에서 강조할 문장

- "읽는 페이지가 아니라, 설득 순서가 있는 랜딩으로 다시 설계했다."

## Nova Admin

### 보여줘야 하는 핵심

- 운영 관점의 정보 구조 설계
- 우선순위 재배치 능력
- 반복 패턴 컴포넌트화 능력

### 좋은 상세 서사

1. 문제
   정보는 많지만 핵심 지표가 먼저 보이지 않음
2. 제약
   운영 화면은 정보량을 줄일 수 없고, 판단 속도만 높여야 함
3. 판단
   상단 지표와 상세 패널의 계층을 분리
4. 구현
   데이터 카드, 차트, 리스트, 필터 패턴을 역할별로 재조합
5. 결과
   시선 이동이 줄고 운영 판단 포인트가 빨리 드러남

### 상세에서 강조할 문장

- "정보를 줄인 것이 아니라, 판단 순서를 다시 설계했다."

## Portfolio

### 보여줘야 하는 핵심

- 자기 서사를 화면 구조로 번역하는 능력
- 인터랙션 구현력
- 반응형 구조 안정성

### 좋은 상세 서사

1. 문제
   이력 나열식 포트폴리오만으로는 프론트엔드 역량 전달이 약함
2. 제약
   인터랙션은 강하지만 읽기성을 해치면 안 됨
3. 판단
   인상은 강하게, 프로젝트 상세는 문서형으로 분리
4. 구현
   스크롤 인터랙션, 테마 분리, 반응형 구조 설계
5. 결과
   첫인상과 실무형 서술이 함께 보이는 구조 확보

## 데이터 모델 개선 제안

현재 `PortfolioProjectContent`에는 아래 필드를 추가하는 것이 좋다.

```ts
interface PortfolioProjectContent {
  ...
  summaryLine?: string;
  constraints?: string[];
  decisions?: string[];
  outcomes?: string[];
  learnings?: string[];
}
```

### 필드 의미

- `summaryLine`
  프로젝트를 한 문장으로 설명하는 문장
- `constraints`
  프로젝트 제약 조건
- `decisions`
  내가 내린 핵심 판단
- `outcomes`
  결과 또는 개선점
- `learnings`
  개인적 학습 포인트

이렇게 되면 프로젝트 상세를 "문제-해결-구현"만이 아니라 "문제-제약-판단-구현-결과" 구조로 확장할 수 있다.

## 디자인 원칙

상세 페이지 디자인은 아래 원칙으로 간다.

### 유지

- 상단 hero 이미지
- 요약 카드
- 다크톤 상세 배경
- 프로젝트별 무드 차이

### 강화

- 요약 카드의 정보 밀도
- 문제/제약/판단의 계층 분리
- 본문 가독성
- 소제목의 의미 명확화

### 줄일 것

- 과도한 전환 효과
- 읽기 도중 방해되는 애니메이션
- 섹션마다 다른 과한 스타일 실험
